# Урок 5. Конвейер: минимальный пайплайн триажа

Уроки 1–4 — про то, *что* нужно для диагноза. Этот урок — про то, как собрать
из этого повторяемый процесс. Анализатор в
[`analyzer/analyzer.py`](../analyzer/analyzer.py) — это **минимальный конвейер**:
самая короткая цепочка стадий, без которой осмысленный ИИ-триаж не работает.
Всё, что сверх неё, — наращивание (урок 7), всё, что меньше, — уже «кинуть
трейсбек в чат».

## Шесть стадий

```
report → parse → enrich → triage → group → export
```

| Стадия | Что делает | Кто делает |
|---|---|---|
| **parse** | читает падения из junit.xml или allure-results | код |
| **enrich** | добирает код теста, код фикстуры, контракт из спеки | код |
| **triage** | ставит диагноз по схеме | **модель** |
| **validate** | отбраковывает ответы не по схеме (внутри triage) | код |
| **group** | склеивает одинаковые падения в один инцидент | код |
| **export** | сводка для людей + JSON для машин + черновики багов | код |

Заметьте пропорцию: модель занимает **одну** стадию из шести. Это и есть
главный архитектурный принцип.

## Разделение труда: код — для гарантий, модель — для смысла

| Код (гарантии) | Модель (смысл) |
|---|---|
| парсинг отчёта | категория падения |
| поиск кода теста в репозитории | гипотеза первопричины |
| контракты из снапшота спеки | улики |
| группировка по фингерпринту | рекомендация действия |
| валидация схемы, заведение багов | заголовок черновика бага |

Правило простое: **всё, что обязано работать одинаково каждый раз, делает код;
всё, что требует понимания смысла, делает модель**. Модель не парсит XML не
потому, что не умеет, а потому что парсинг должен быть гарантированным, а не
«скорее всего правильным».

## Валидация: схема — граница ответственности

JSON-режим API (`response_format={"type": "json_object"}`) гарантирует только
*синтаксис* JSON. Нашу *схему* — набор полей, enum категорий, непустые улики —
гарантирует валидация Pydantic на нашей стороне
([`analyzer/schema.py`](../analyzer/schema.py)):

```python
class Category(str, Enum):
    PRODUCT = "продукт"
    TEST_CODE = "код_автотеста"
    INFRASTRUCTURE = "инфраструктура"
    TOOLING = "инструментарий"
    FLAKY = "flaky"

class Verdict(BaseModel):
    test: str
    category: Category                       # только из enum
    confidence: float = Field(ge=0, le=1)
    root_cause: str
    evidence: str                            # вердикт без улик не принимается
    suggested_action: str
    needs_human: bool
    draft_bug_title: str | None = None
```

Ответ не прошёл валидацию → модели показывают её ошибку и просят строго
валидный JSON, одна повторная попытка. Не помогло — падение честно уходит
в ручной разбор. Никогда не подгоняйте кривой ответ руками.

Ещё две настройки вызова, важные для триажа: `temperature: 0` (диагноз, не
творчество — нам нужна воспроизводимость, а не разнообразие) и отдельный вызов
модели на каждое падение (одно падение — один диагноз, контексты не
перемешиваются).

## Группировка: одна причина — один инцидент

Склейку одинаковых падений делает **код, не модель** — детерминированный
фингерпринт:

```python
def fingerprint(f: Failure, v: Verdict) -> str:
    line = f.message.splitlines()[0]
    normalized = re.sub(r"[0-9a-f]{8,}|\d+", "N", line.lower())
    return f"{v.category.value}|{normalized[:120]}"
```

Категория + первая строка сообщения, где числа и хэши заменены на `N` (иначе
`id=123` и `id=456` считались бы разными причинами). Четыре теста, упавшие
из-за одной фикстуры, слипаются в один инцидент **гарантированно**, а не
«скорее всего» — потому что группирует код. Попроси мы модель «сгруппируй
похожие» — получили бы каждый раз немного другую разбивку.

## Export: три артефакта под три аудитории

- `triage.md` / `triage.html` — сводка для людей: к стендапу, в чат;
- `verdicts.json` — вердикты для машин: метрики, дашборды, история;
- `issues.json` — черновики багов (о них урок 6).

Один прогон — одна сводка: «12 красных → 9 причин, вот улики по каждой».
Это и есть ответ на утренний вопрос «что сломалось?».

## Экономика: почему это вообще выгодно

Порядок цифр: 12 падений × 2–4 тыс. токенов ≈ 30–50 тыс. токенов на прогон —
копейки против часа инженера. Оптимизации очевидны: типовые падения — дешёвой
или локальной моделью, спорные — флагманской (флаг `--escalate` в анализаторе);
системный промпт стабилен и кешируется провайдером; ночной регресс не
интерактивен — можно гнать батчем.

## Почему именно «минимальный»

Выкиньте любую стадию — и конструкция разваливается:

- без **parse** — нечего разбирать;
- без **enrich** — модель гадает по одной строке (урок 3);
- без **validate** — вердикты нельзя обрабатывать кодом, всё снова руками;
- без **group** — 30 падений остаются 30 задачами вместо 9;
- без **export** — результат не доходит ни до людей, ни до трекера.

При этом в конвейере нет ничего лишнего: ни базы данных, ни истории прогонов,
ни RAG, ни дообучения. Он работает от одного файла отчёта — живой API не
обязателен. Это осознанно: сначала минимальное ядро, которое приносит пользу
с первого прогона, потом наращивание под нужды проекта (урок 7).

## ✅ Как надо / ❌ как не надо

| ❌ Как не надо | ✅ Как надо |
|---|---|
| Просить модель «разбери весь отчёт целиком» | Одно падение — один вызов — один вердикт |
| Доверять модели парсинг, группировку, заведение багов | Код — для гарантий, модель — только для смысла |
| Верить, что JSON-режим гарантирует схему | Валидировать схему на своей стороне |
| Подправлять кривой ответ модели руками | Ретрай с показом ошибки; не вышло — ручной разбор |
| temperature побольше, «чтобы модель подумала» | temperature 0: триаж — не творчество |
| Группировать «на глаз» или силами модели | Детерминированный фингерпринт в коде |
