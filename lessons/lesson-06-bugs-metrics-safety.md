# Урок 6. Автозаведение багов, метрики и границы автоматизации

Конвейер поставил диагнозы и сгруппировал причины. Последний соблазн — «пусть
он сам и баги заведёт». Можно. Но именно здесь проходит граница, на которой
ИИ-триаж либо остаётся полезным, либо превращается в генератор мусора в трекере.

## Из вердикта — черновик бага

Маппинг тривиален, его делает код:

```json
{
  "title": "GET /practice/temperature: неверная формула конвертации при celsius=100",
  "category": "продукт",
  "body": "Затронутые тесты: ...\n\nПервопричина: ...\n\nУлики: ...\n\nРекомендация: ...",
  "labels": ["ai-triage", "продукт"],
  "status": "draft"
}
```

Три обязательных атрибута автоматики:

- **status: draft** — это черновик, а не готовый баг;
- **метка ai-triage** — всегда видно, что карточку готовила машина;
- **фильтр на входе** — спорное в трекер не уходит:

```python
if v.needs_human or v.confidence < 0.6:
    continue  # спорное — человеку, не в трекер
```

## Human-in-the-loop: заведение выключено по умолчанию

Анализатор по умолчанию **только пишет файлы**. Цикл выглядит так:

```
модель ставит вердикты → инженер читает triage.md →
правит / отклоняет спорные → --file-issues заводит принятые
```

Автозаведение — **осознанное действие после взгляда человека**, а не побочный
эффект прогона. На тренажёре черновики уходят в `POST /issues` мини-трекера
[BookShelf](https://qahacking.up.railway.app/issues); в реальном проекте endpoint
меняется на Jira / YouTrack — код тот же, меняется маппинг полей. `category`
при этом решает маршрутизацию: продукт → доска продукта, инфраструктура →
DevOps, инструментарий → QA-обвязка, flaky → карантин.

## Метрики: только парами

Любая одиночная метрика ИИ-триажа легко «улучшается» во вред делу. Поэтому —
парами, где вторая метрика держит первую:

| Метрика | ↔ | Противовес |
|---|---|---|
| **Скорость**: время «отчёт → сводка» | ↔ | **Качество**: точность триажа против ручных вердиктов |
| **Автоматизация**: доля багов, заведённых без правок | ↔ | **Шум**: доля мусорных черновиков |

Рабочий ориентир приёмки: **60–80% вердиктов принимаются без правок**.
Выше — подозрительно (вы, скорее всего, не ревьюите, а штампуете «ок»);
ниже — чините вход, то есть пакет контекста.

И замыкающая метрика, ради которой всё затевалось: **пропущенные в прод баги
по прогонам с ИИ-триажем не должны расти**. Если сводки стали быстрее, а баги
стали проскакивать — триаж не работает, он маскирует.

## Анализатор ошибся. Что делать?

Алгоритм принятия решения — в порядке применения:

1. **Не хватило контекста?** (не было контракта, кода теста, кода фикстуры)
   → чинить enrich — стадию сбора контекста.
2. **Это повторяющееся правило?** (например, «место ошибки определяет категорию»)
   → добавить правило в промпт; изменение пройдёт ревью, как код.
3. **Редкий, но повторяющийся кейс?** → сохранить исправленный пример в базу
   знаний: сначала как few-shot-пример в промпте, позже — подключить поиском.
4. **Разовый или спорный случай?** → ничего не менять: для этого есть
   `needs_human`.

> **Не лечите модель первой. Сначала лечите входные данные и правила.**
> Смена модели на «умнее» и дообучение — последние средства, а не первые.
> Fine-tuning имеет смысл, только когда накоплены тысячи размеченных примеров.

## Безопасность: что течёт в логи

В трейсбеках и stdout живут: Bearer-токены из заголовков, e-mail и персональные
данные из тел ответов, внутренние URL. Отправляя пакет контекста во внешнюю
модель, вы отправляете туда и это.

Правило: **маскировать до отправки в модель** (регексами на стадии сборки
контекста) — или держать модель в своём контуре (локальная / self-hosted;
анализатор работает с любым OpenAI-совместимым endpoint, включая Ollama и
LM Studio — см. README).

## ✅ Как надо / ❌ как не надо

| ❌ Как не надо | ✅ Как надо |
|---|---|
| Автозаведение багов как побочный эффект прогона | Заведение выключено по умолчанию; сначала человек читает сводку |
| Баги от ИИ неотличимы от багов людей | status: draft + метка ai-triage |
| Спорные вердикты тоже уходят в трекер | needs_human и confidence < 0.6 — только человеку |
| Мерить один показатель («скорость разбора выросла!») | Метрики парами + замыкающая: пропущенные баги |
| При ошибке сразу менять модель или дообучать | Сначала контекст, потом правила, потом база знаний |
| Слать сырые логи с токенами и ПД во внешний API | Маскировать до отправки или держать модель в контуре |
