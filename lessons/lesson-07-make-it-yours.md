# Урок 7. Свой триаж под свой проект: что крутить и куда расти

Конвейер из урока 5 **переносим**: меняются вход и правила, ядро остаётся тем
же. Этот урок — карта настройки под ваш проект и маршруты роста, когда
минимального пайплайна станет мало.

## Пять точек настройки

| Точка | Что заменить | Где это в примере |
|---|---|---|
| **enrich** | ваш репозиторий тестов + ваш источник контрактов (OpenAPI-снапшот, требования, acceptance criteria) | `find_test_source`, `find_fixture_source`, `find_contracts` в [`analyzer/analyzer.py`](../analyzer/analyzer.py) |
| **таксономия** | категории под ваш стек: UI-тестам нужны «селектор устарел» и «тайминги/анимации», мобильным — «девайс/эмулятор» | enum `Category` в [`analyzer/schema.py`](../analyzer/schema.py) + описания в промпте |
| **промпт** | ваши правила разбора и якорь-примеры типичных падений | [`analyzer/prompt.md`](../analyzer/prompt.md) |
| **фингерпринт** | регекс нормализации под формат *ваших* сообщений об ошибках | `fingerprint()` в analyzer.py |
| **провайдер** | любой OpenAI-совместимый endpoint или локальная модель — через переменные окружения, без правки кода | `.env` (см. README) |

Порядок важен: **сначала enrich и промпт, потом всё остальное** — они дают
основной прирост качества (урок 3: ~70% решает вход).

Правила переноса те же, что были во всех уроках: таксономия остаётся закрытым
списком с действиями; каждое новое правило в промпте — из реальной ошибки
разбора, через ревью; фингерпринт остаётся детерминированным кодом.

## С чего начать на своём проекте

1. **Уровень 0, вручную.** Возьмите промпт, адаптируйте таксономию под свой
   стек, скормите в чат 5–10 реальных падений. Посчитайте, сколько вердиктов
   вы приняли бы без правок. Меньше половины — чините пакет контекста, а не
   автоматизируйте.
2. **Соберите enrich.** Научитесь кодом доставать код упавшего теста и контракт
   задетой функциональности. Это самая проектно-зависимая часть — и самая ценная.
3. **Заведите схему и валидацию.** С этого момента вердикты можно копить,
   считать и сравнивать.
4. **Неделю живите в режиме «файлы, не баги».** Сводка к стендапу — да,
   автозаведение — нет. Меряйте долю принятых без правок.
5. Только потом — черновики в трекер (draft + метка, урок 6).

## Куда это растёт

Минимальный конвейер — фундамент. Дальше, по мере созревания:

### 1. История прогонов → честный flaky
Самое ценное расширение. Храните `verdicts.json` каждого прогона — и у модели
появляется вход «этот тест падает 3-й раз подряд» vs «первый раз за месяц».
Без истории flaky отличается от стабильного падения только догадкой; таблица
уровней входа из урока 3 получает свою верхнюю ступень.

### 2. CI-гейт
Триаж как шаг пайплайна после регресса: упало — прямо в CI появляется сводка
«N красных → M причин, из них продукт: K». Стендап начинается с диагнозов,
а не с открывания вкладок.

### 3. База знаний и поиск по прошлым багам
Исправленные вердикты копятся → сначала это few-shot-примеры в промпте, потом
поиск похожих случаев (RAG): «такое падение мы уже видели в BUG-1234». Дежавю
инженера-старожила, только автоматическое.

### 4. Жирный вход
Переход с junit.xml на allure-results: модель видит шаги и реальные тела
запросов/ответов. Требует дисциплины разметки тестов (урок 3: кривой Allure =
уверенный мусор на входе).

### 5. Двухуровневые модели
Типовые падения — дешёвой/локальной моделью, спорные — флагманской
(в анализаторе это флаг `--escalate`). Экономика: качество там, где оно нужно,
копейки — где не нужно.

### 6. Интеграция с трекером
`POST /issues` тренажёра → Jira/YouTrack REST API. Код тот же, меняются endpoint
и маппинг полей; `category` маршрутизирует карточку на нужную доску.

Чего в этом списке **нет**: дообучения модели. Оно в самом конце дороги
(урок 6: сначала контекст, потом правила, потом база знаний), когда накоплены
тысячи размеченных вердиктов.

## Чек-лист зрелости ИИ-триажа

- [ ] Таксономия — закрытый список, у каждой категории есть действие
- [ ] Пакет контекста: трейсбек + код теста + контракт
- [ ] Промпт — файл в репозитории, правки через ревью
- [ ] Ответ модели валидируется схемой на нашей стороне
- [ ] Группировку делает детерминированный код
- [ ] Вердикт без улик не принимается; confidence ≠ истина
- [ ] needs_human — легальный исход; спорное не попадает в трекер
- [ ] Баги — только черновики с меткой, после взгляда человека
- [ ] Метрики парами; следим за пропущенными в прод багами
- [ ] Секреты и ПД маскируются до отправки в модель

## ✅ Как надо / ❌ как не надо

| ❌ Как не надо | ✅ Как надо |
|---|---|
| Скопировать чужой промпт и таксономию как есть | Таксономия и правила — под ваш стек и ваши падения |
| Начинать с автозаведения багов и CI-гейта | Начинать с уровня 0 и доли принятых вердиктов |
| «Модель слабая, нужен fine-tuning» | Сначала enrich, потом правила, потом база знаний |
| Строить сразу платформу с историей и RAG | Минимальное ядро → расти по мере созревания |
| Привязаться к одному провайдеру модели | OpenAI-совместимый клиент: провайдер меняется через env |
